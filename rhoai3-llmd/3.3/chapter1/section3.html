Cluster Bootstrapping

Before you can build the factory line, you must pour the concrete.

If you hand a Platform Engineer a brand new OpenShift 4.20 cluster with the Red Hat OpenShift AI (RHOAI) Operator installed, they cannot immediately deploy an LLMInferenceService.

Distributed inference is a deeply integrated architecture that requires cluster-level components to manage multi-pod scaling and advanced network intercepts. Before developers can use the GitOps fast-track scripts, a Cluster Administrator must perform this one-time bootstrapping phase.

1. The Leader Worker Set (LWS) Operator

LLM inference often requires deploying a model across multiple GPUs, or even multiple physical nodes. Kubernetes natively uses Deployments or StatefulSets, but these are not optimized for distributed machine learning workloads where a "leader" pod must tightly coordinate with "worker" pods.

  • The Requirement: You must install the LeaderWorkerSet Operator from Ecosystem (OperatorHub).

  • The "Why": llm-d uses LWS under the hood to deploy the vLLM engine. It ensures that if one worker pod in a distributed inference group fails, the entire set is safely restarted to prevent split-brain execution and corrupted memory states.

2. Activating the Distributed Workloads Components

By default, a standard RHOAI installation might only enable basic Jupyter workspaces and pipelines. We must explicitly turn on the engines required for our AI Factory.

  • The Requirement: Edit the DataScienceCluster (DSC) custom resource.

  • The "Why": The KServe and distributed workload controllers do not consume cluster resources unless explicitly requested.

Ensure your DSC YAML includes the following managed states:

spec:
  components:
    kserve:
      managementState: Managed
    distributedworkloads:
      managementState: Managed

3. The Gateway API (The Smart Front Door)

This is the most critical networking concept in the llm-d architecture.

In traditional OpenShift deployments, you create a Route that points directly to a Kubernetes Service. The standard ingress controller simply passes the traffic through as quickly as possible.

This does not work for intelligent routing. If we want the Inference Scheduler (the "Brain") to score queue lengths and check KV cache affinity, we need a network component capable of intercepting the HTTP request, pausing it, asking the Scheduler for directions, and then forwarding the payload.

  • The Requirement: We must explicitly deploy an Envoy-based Gateway API resource named openshift-ai-inference in the openshift-ingress namespace.

  • The "Why": The standard OpenShift router is too basic for this task. The Gateway API provides the advanced traffic interception required by the llm-d control plane.

4. Securing the Gateway (Connectivity Link)

In our upcoming deployment labs, we will deploy models with the annotation security.opendatahub.io/enable-auth: 'false' to focus purely on the routing mechanics. However, an enterprise AI Factory cannot leave endpoints exposed.

  • The Requirement: If you intend to secure your endpoints and enable Models-as-a-Service (MaaS) API rate limiting, the cluster administrator must install the Red Hat Connectivity Link operator (which provides Kuadrant and Authorino).

  • The "Why": llm-d does not handle its own authentication. If you flip enable-auth to 'true', the Gateway relies entirely on Kuadrant to validate JSON Web Tokens (JWTs) and enforce usage tiers. Without Connectivity Link installed, secured routing will fail.

Deploying the Gateway

A cluster admin only needs to apply this configuration once per cluster:

apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: openshift-ai-inference
  namespace: openshift-ingress
spec:
  gatewayClassName: openshift-default-gateway-class
  listeners:
  - allowedRoutes:
      namespaces:
        from: All
    name: http
    port: 80
    protocol: HTTP
  - allowedRoutes:
      namespaces:
        from: All
    name: https
    port: 443
    protocol: HTTPS

The Gateway & Models-as-a-Service (MaaS)

Taking the time to configure this Gateway unlocks one of the most powerful paradigms in enterprise AI: Models-as-a-Service (MaaS).

If your organization is hosting 50 different models (Granite, Llama, Qwen, etc.) for various business units, you do not want to manage 50 completely different ingress URLs, API tokens, and security policies.

Because we use the centralized openshift-ai-inference Gateway, it acts as the single, unified API endpoint for the entire enterprise.

  • How it scales: As Platform Engineers deploy new LLMInferenceService resources in different namespaces, the KServe operator automatically creates HTTPRoute objects. These routes dynamically attach to our single Gateway.

  • The Value: A developer simply points their application to https://<gateway-url>/<namespace>/<model-name>/v1/chat/completions.

  • The MaaS Advantage: By funneling all AI traffic through this single Gateway, Platform Architects can easily implement centralized rate-limiting, API key authentication (via TrustyAI or 3scale), and accurate chargeback billing per business unit based on token consumption.

With the concrete poured and the Gateway API standing ready, the cluster is fully prepared. Next, we will step into the shoes of the Platform Engineer and execute the GitOps deployment lab.
