*A factory requires different tools for different production lines. llm-d provides the complete toolkit.*

In the previous module, we established *why* the business needs distributed inference: to drive down costs and guarantee performance. But how exactly does `llm-d` achieve this state-of-the-art performance on Kubernetes? 

`llm-d` is not a single feature; it is a comprehensive suite of optimizations designed to squeeze every ounce of value out of modern hardware accelerators. As a Platform Engineer, you must understand the available tools in this framework to design the right architecture for your specific Compound AI Systems.

Here are the five core capabilities of Red Hat OpenShift AI 3.3's distributed inference stack, and the strategic design decisions around when to use them.

== 1. Intelligent Inference Scheduling (Cache-Aware Routing)

This is the bread and butter of `llm-d`. Instead of a standard Kubernetes ingress controller blindly round-robining traffic, `llm-d` deploys vLLM behind a smart Envoy load balancer. 

* **How it works:** The scheduler actively monitors the cluster. It routes traffic based on utilization (who is least busy), fairness (preventing starvation), and **prefix-cache awareness**. If User A asks a question about a 10-page document, and User B asks a follow-up question about that *same* document, the scheduler knows which specific GPU already has that document loaded in its KV cache and routes User B there instantly.
* **When to use it:** **Always.** This is the foundational capability that enables high-density sharing and turns a standard endpoint into a Model-as-a-Service (MaaS) offering. 

== 2. Disaggregated Serving (Separating Read from Write)

LLM inference consists of two distinct phases: **Prefill** (reading and processing the user's prompt) and **Decode** (generating the response token-by-token). Prefill is highly compute-intensive, while Decode is highly memory-bandwidth intensive.

* **How it works:** Disaggregated serving allows you to physically split these workloads. You assign specific GPUs purely to be "Prefill Servers" and others to be "Decode Servers." The Prefill server reads the massive prompt, calculates the KV cache, and transmits it over a high-speed network (like RoCE) to the Decode server to stream the response.
* **When to use it:** In heavy Retrieval-Augmented Generation (RAG) workloads. If your users are uploading massive legal contracts (which cause huge prefill spikes that block decode generation), separating the phases ensures that heavy readers never interrupt fast writers, dramatically lowering Time-To-First-Token (TTFT).



== 3. Wide Expert-Parallelism (Multi-Node Scaling)

Traditional tensor parallelism shards a model across multiple GPUs *within the same physical server*. But what happens when the model is bigger than the server itself?

* **How it works:** `llm-d` supports distributing massive Mixture-of-Experts (MoE) models across multiple physical nodes. It leverages technologies like NVSHMEM and high-speed network fabrics to allow GPUs on different physical machines to act as a single, unified accelerator brain.
* **When to use it:** When you are deploying "Trillion Parameter" scale models (e.g., massive DeepSeek or Llama 405B variants) that simply exceed the VRAM of a single 8-GPU node, even with quantization. 

== 4. Tiered KV Prefix Caching

The KV cache is the memory the model uses to "remember" the conversation. VRAM (GPU memory) is the fastest, but it is incredibly expensive and limited.

* **How it works:** `llm-d` can intelligently offload KV cache entries from precious GPU VRAM down to cheaper CPU system RAM, or even further down to local NVMe SSDs or remote storage. When that context is needed again, it is pulled back up the tier.
* **When to use it:** In high-concurrency environments where system prompts, standard operating procedures, or large document contexts are frequently reused. This allows you to serve massive effective context windows without constantly hitting "Out of Memory" (OOM) errors on the GPU.

== 5. Workload Autoscaling

Static sizing is dangerous. If you over-provision, you waste money. If you under-provision, your application crashes.

* **How it works:** Integrating seamlessly with the Kubernetes Horizontal Pod Autoscaler (HPA) or customized variant autoscalers, `llm-d` can dynamically scale your vLLM worker pods up and down based on real-time metrics (like queue depth or KV cache utilization).
* **When to use it:** Handling variable enterprise traffic patterns. You can scale your IT Helpdesk copilot down to zero at night, returning those valuable GPUs to the Kueue quota pool so the Data Science team can use them for overnight model training, and scale back up in the morning before business hours.
